# Customization - 三模型定制

## 作用

Customization 采用三模型实现：

- `Metadata Customization`：面向元数据对象（Field/Attribute）的租户级定制
- `Client Customization`：面向 Client 的租户级定制
- `Layout Profile Derivation`：面向 Layout 的方案派生定制（Tenant/User）

该设计用于解决“Client 定制偏行为覆盖、Layout 定制偏方案派生”的差异，避免强行统一为单一机制。

## 模型划分

| 模型 | 作用域 | 目标对象 | 主要定位键 |
|-----|-----|-----|-----|
| Metadata Customization | Tenant | field, attribute | tenantId + entityCode + targetType + targetCode |
| Client Customization | Tenant | client | tenantId + entityCode |
| Layout Profile Derivation | Tenant/User | layout profile + layouts | scope + owner + profile.code（Profile） + entityCode + profileId + type（Layout） |

## 一、Metadata Customization

### 核心字段

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|-----|------|------|-------|------|
| tenantId | BIGINT | 是 | - | 租户ID |
| entityCode | VARCHAR(100) | 是 | - | 关联实体编码（Entity.code） |
| targetType | VARCHAR(50) | 是 | - | 目标类型（field/attribute） |
| targetCode | VARCHAR(100) | 是 | - | 目标编码（Field.code/Attribute.code） |
| properties | JSON | 是 | - | 要覆盖的属性对象 |
| yn | CHAR(1) | 是 | N | 软删除标记 |

### 支持覆盖

- `field`：`name`, `required`, `readOnly`, `status`
- `attribute`：`name`, `required`, `status`

### 约束

1. `tenantId + entityCode + targetType + targetCode + yn` 必须唯一
2. `targetCode` 必须在对应实体中存在
3. `entityCode` 必须指向有效 `Entity.code`
4. `entityCode` 为只读关联键，不允许通过定制流程修改
5. 实体与目标对象 `customizable` 都必须为 `true`
6. 覆盖 `status` 时必须是 `ACTIVE/DISABLED/DRAFT`

## 二、Client Customization（Tenant）

### 核心字段

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|-----|------|------|-------|------|
| tenantId | BIGINT | 是 | - | 租户ID |
| entityCode | VARCHAR(100) | 是 | - | 关联实体编码（Entity.code） |
| properties | JSON | 是 | - | Client 覆盖配置 |
| yn | CHAR(1) | 是 | N | 软删除标记 |

### 支持覆盖

- `controller`
- `views`, `recordViews`, `relationshipPanels`
- 其他 Client 白名单行为字段

### 约束

1. `tenantId + entityCode + yn` 必须唯一
2. 仅允许从 `scope=PLATFORM` 的 Client 基线覆盖
3. `entityCode` 必须指向有效 `Entity.code`
4. `entityCode` 为只读关联键，不允许通过定制流程修改
5. 实体与 Client 对象 `customizable` 都必须为 `true`
6. 不能覆盖结构标识字段（如 `scope`, `code`, `entityCode`）

## 三、Layout Profile Derivation（Tenant/User）

### 核心思想

- Layout 定制统一为“派生一套独立 Profile”
- 小改和大改都通过同一机制处理
- 不再维护 Layout patch 记录
- 派生采用按需模式：仅派生 Profile 元数据；Layout 条目按实体/类型在发生修改时创建

### 归属模型（scope + tenantId + owner）

| scope | tenantId | owner 含义 | 示例 |
|-----|-----|-----|-----|
| PLATFORM | 空 | 固定 `platform` | `tenantId=null, owner=platform` |
| TENANT | 必填 | 租户标识（等于 `tenantId`） | `tenantId=1001, owner=1001` |
| USER | 必填 | 用户标识 | `tenantId=1001, owner=501` |

补充约束：

1. `scope=PLATFORM` 时 `tenantId` 为空，且 `owner=platform`
2. `scope=TENANT` 时 `tenantId` 必填，且 `owner=tenantId`
3. `scope=USER` 时 `tenantId` 必填，`owner` 必须是有效 `userId`（`userId` 全局唯一），且该用户属于 `tenantId`
4. `scope + owner + profile.code` 必须唯一（业务唯一键，不包含 `yn`）
5. 同一 `scope+owner` 下最多允许 1 条 `status=ACTIVE` 的 Profile
6. 仅允许从 `status=ACTIVE and customizable=true` 的源 Profile 派生
7. `scope=USER` 的 Profile 视为终态，禁止继续派生

### profile.code 与派生关系

- `profile.code`：方案编码，在同一 `scope+owner` 内唯一
- 运行时解析：先按 `scope+owner+profile.code` 选中 LayoutProfile，再按 `entityCode+profileId+type` 加载 Layout

### 按需派生规则

1. 创建派生时仅创建 `LayoutProfile` 记录，不自动复制所有实体 Layout
2. 当某实体某类型发生修改时，才创建或更新对应 `Layout(entityCode+profileId+type)`
3. 未修改的实体布局在运行时按 `USER > TENANT > PLATFORM` 顺序回退读取

### 定制对象范围

Profile 派生后，可在该 Profile 下修改：

- Layout 定义（list/detail/form/relationship）

其中 `relationship` 布局为聚合配置：按 `entityCode + profileId + type=relationship` 定位，在 `layout.relationshipLayouts` 中按 `linkCode` 分组定义各关系面板。

## 四、运行时合并

### 合并优先级

- Metadata：`Platform Metadata < Metadata Customization`
- Client：`Platform Client < Client Customization`
- Layout：`Platform Profile < Tenant Profile < User Profile`

### 合并流程

1. 校验实体 `status=ACTIVE`
2. 应用 `Metadata Customization`，并过滤 `status=DISABLED` 的字段/属性
3. 应用 `Client Customization`（若存在）
4. 对每个 `(entityCode, type)` 按 `USER -> TENANT -> PLATFORM` 逐层处理：
   - 先在当前层解析 `LayoutProfile`：优先显式 `profile.code`，未显式指定则取该层 `status=ACTIVE` 的 Profile
   - 再用该 `profileId + entityCode + type` 查找 Layout
   - 当前层无可用 Profile 或无对应 Layout 时，继续回退下一层
5. 命中首个可用 Layout 后停止回退，作为该 `(entityCode, type)` 的最终布局
6. 若三层都未命中可用 Layout，则返回空布局（不做默认 `profile.code` 隐式回退）
7. 输出最终运行时元数据视图

### 实体归属强约束（统一规则）

1. `Field`、`Attribute`、`Client`、`Layout` 必须显式包含 `entityCode`
2. `LayoutProfile` 不绑定 `entityCode`，仅定义作用域维度下的方案模板
3. 运行时 Layout 必须与当前 `entityCode` 一致，否则拒绝加载并记录审计日志
4. 任何跨实体引用（如 `Layout` 使用不存在于当前实体的 field/link）必须在发布前校验失败
5. `Entity.code` 创建后不可变，`entityCode` 仅允许通过迁移流程切换，不允许在线直接改写

### Layout 双闸门判定（避免分歧）

1. 先判定 `LayoutProfile.customizable`（是否允许派生下一层 Profile）
2. 仅在第 1 步通过后，判定 `Layout.customizable`（是否允许修改具体 Layout 条目）
3. 拒绝优先（deny-first）：任一层为 `false`，拒绝对应变更
4. `scope=USER` 的 `LayoutProfile` 固定为终态（`customizable=false`）

判定伪代码：

```text
if !sourceProfile.customizable:
  deny("profile not derivable")

createOrSelectDerivedProfile()

if !layoutItem.customizable:
  deny("layout item is read-only in derived profile")

allowLayoutMutation()
```

## 五、id 与 code 使用准则

### 使用 `id` 的场景

- 数据库主外键关联（如 `tenantId`, `userId`）
- 内部 join 与性能敏感查询
- 审计与版本快照定位

### 使用 `code` 的场景

- API 与配置对外定位（`entityCode`, `profile.code`, `targetCode`）
- 跨环境可迁移配置（dev/stage/prod）
- 人可读、可维护的业务规则

### 原则

- 对外定位用 `code`
- 对内关联用 `id`
- 服务层负责 `code -> id` 解析与校验

## 六、示例

### Metadata Customization：禁用字段

```json
{
  "tenantId": 1001,
  "entityCode": "Product",
  "targetType": "field",
  "targetCode": "internalNotes",
  "properties": {
    "status": "DISABLED"
  }
}
```

### Client Customization：租户覆盖列表视图

```json
{
  "tenantId": 1001,
  "entityCode": "Product",
  "properties": {
    "views": {
      "list": "views/record/list-tenant"
    }
  }
}
```

### Tenant Profile：从平台方案派生

```json
{
  "scope": "TENANT",
  "tenantId": 1001,
  "owner": "1001",
  "code": "tenant-default",
  "status": "ACTIVE"
}
```

### User Profile：用户专属方案

```json
{
  "scope": "USER",
  "tenantId": 1001,
  "owner": "501",
  "code": "user-compact",
  "status": "ACTIVE"
}
```
