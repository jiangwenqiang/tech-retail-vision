# SaaS 元数据管理系统

基于元数据模型的完整实现，支持平台基线和租户级别的元数据管理。每个对象采用最小化实现，预留扩展空间。

## 基线与租户管理机制

### 作用域规则

| 作用域   | 定义者     | 可见性   | 可修改性     |
| -------- | ---------- | -------- | ------------ |
| PLATFORM | 平台管理员 | 所有租户 | 仅平台管理员 |
| TENANT   | 租户管理员 | 当前租户 | 租户管理员   |
| USER     | 普通用户   | 当前用户 | 用户本人     |

### Customization - 三模型定制

定制能力采用三模型：

- **Metadata Customization**：用于 Field/Attribute 的元数据定制
- **Client Customization**：用于 Client 的租户级定制（Tenant）
- **Layout Profile Derivation**：用于 Layout 的方案派生定制（Tenant/User）

详细说明见：[07.customization.md](./07.customization.md)

**核心设计**：

- **分层治理**：元数据定制与视图定制分离，避免 profile 维度与元数据规则耦合
- **作用域分级**：Client 采用 PLATFORM/TENANT 两层；LayoutProfile 采用 PLATFORM/TENANT/USER 三层
- **状态一致**：实体/字段/属性通过 `status` 统一控制启用/禁用
- **可扩展**：支持按 `targetType` 扩展目标对象与可覆盖属性
- **权限控制**：只能定制 `customizable=true` 的元数据对象
- **定位策略**：对外以 `code` 定位，对内以 `id` 关联

### 可见性管理

| 元数据对象        | 平台级可见性控制 | 租户级可见性控制                         | 说明                                                   |
| ----------------- | ---------------- | ---------------------------------------- | ------------------------------------------------------ |
| Entity Definition | status + scope   | -                                        | 只有 status=ACTIVE 的 PLATFORM 实体才对所有租户可见    |
| Field             | status           | Metadata Customization（覆盖 status）    | 覆盖 status 为 DISABLED 来禁用可定制字段               |
| Attribute         | status           | Metadata Customization（覆盖 status）    | 覆盖 status 为 DISABLED 来禁用可定制属性               |
| Client Definition | -                | Client Customization（Tenant）           | 允许租户覆盖客户端行为配置                             |
| Layout            | -                | Layout Profile Derivation（Tenant/User） | 基于 LayoutProfile 派生，按 `scope+owner` 选择生效方案 |

**可见性规则**：

- **实体可见性**：通过 `status` 和 `scope` 字段控制，只有 `status=ACTIVE` 的 PLATFORM 实体才对所有租户可见
- **字段/属性可见性**：通过 `status` 字段控制，租户可通过 Metadata Customization 覆盖 status 为 DISABLED 来禁用可定制元数据
- **Client 定制**：通过 Tenant Client 覆盖平台 Client 配置
- **Layout 定制**：统一通过 LayoutProfile 派生处理，不区分微调和大改
- **Layout 派生策略**：按需派生；先派生 Profile，Layout 条目按实体/类型在实际变更时落库
- **布局方案维度**：LayoutProfile 作为方案模板独立定义，不绑定实体
- **Layout 归属键**：Layout 通过 `entityCode + profileId` 绑定具体实体与具体 LayoutProfile
- **状态一致性**：实体/字段/属性使用统一的状态管理机制（ACTIVE/DISABLED/DRAFT）
- **Client/Layout 生效**：Client/Layout 不独立维护 `status`，由关联实体状态与选中配置共同决定生效
- **配置优先级**：
  - Client：Platform Client < Tenant Client
  - Layout：Platform Profile < Tenant Profile < User Profile
- **运行时合并**：先合并元数据层，再合并 Client；Layout 按 `USER -> TENANT -> PLATFORM` 逐层处理，在各层已启用 Profile 下优先命中同维度 `Layout(entityCode+profileId+type)`，三层均无可用 Layout 时返回空布局

### 实体归属链（强一致）

- `Field`、`Link`、`Index`、`Attribute`、`Client`、`Layout` 均必须显式绑定 `entityCode`
- `LayoutProfile` 不绑定 `entityCode`，按 `scope+owner` 维度定义方案模板
- 运行时同一元数据视图中的对象必须满足同一 `entityCode`
- `entityCode` 必须引用有效 `Entity.code`（`Entity.code` 全局唯一）
- `Entity.code` 创建后不可变；禁止直接改码导致级联失配，必须通过迁移流程切换
- 关键唯一键：
  - `Field`: `entityCode + code + yn`
  - `Link`: `entityCode + code + yn`
  - `Index`: `entityCode + code + yn`
  - `Attribute`: `entityCode + code + yn`
  - `Client`: `entityCode + scope + yn`
  - `LayoutProfile`: `scope + owner + code`（业务唯一键，不包含 `yn`）
  - `Layout`: `entityCode + profileId + type`（业务唯一键，不包含 `yn`）

### 定制化管理

| 元数据对象        | 平台级控制        | 租户级定制                               | 说明                          |
| ----------------- | ----------------- | ---------------------------------------- | ----------------------------- |
| Entity            | customizable 字段 | -                                        | 控制是否允许租户定制该实体    |
| Field             | customizable 字段 | Metadata Customization（属性覆盖）       | 只能定制标记为可定制的字段    |
| Attribute         | customizable 字段 | Metadata Customization（属性覆盖）       | 只能定制标记为可定制的属性    |
| Attribute（新增） | -                 | scope=TENANT                             | 租户可添加自定义属性          |
| Client Definition | customizable 字段 | Client Customization（Tenant）           | 通过租户级覆盖定制客户端配置  |
| Layout            | customizable 字段 | Layout Profile Derivation（Tenant/User） | 通过派生 Profile 定制布局配置 |

**定制化规则**：

- 所有可定制对象由 `customizable` 控制定制权限
- Metadata Customization 仅作用于 Field/Attribute
- Client Customization 仅作用于 Client（Tenant）
- Layout Profile Derivation 仅作用于 Layout（Tenant/User）
- Layout 运行时按层级优先级选择 Profile：按 `USER -> TENANT -> PLATFORM` 逐层处理

**`customizable` 统一语义**：

- `customizable=true`：允许“下一层 scope”基于该对象做定制或派生
- `customizable=false`：禁止下层 scope 定制，必须沿用当前对象定义
- 对 `Client(scope=PLATFORM)`：控制 Tenant 是否可覆盖
- 对 `LayoutProfile(scope=PLATFORM)`：控制 Tenant 是否可派生
- 对 `LayoutProfile(scope=TENANT)`：控制该租户下 User 是否可派生
- 对 `LayoutProfile(scope=USER)`：视为终态，不再向下派生（固定 `customizable=false`）
- 对 Layout 定制：采用双闸门，先判定 `LayoutProfile.customizable`（可派生），再判定 `Layout.customizable`（可改条目），任一为 `false` 即拒绝

### 覆盖规则

**Metadata Customization 允许覆盖**（需 `customizable=true`）：

- Field：name、required、readOnly、status
- Attribute：name、required、status

**Client Customization 允许覆盖**：

- Client：可覆盖客户端行为配置白名单属性

**Layout Profile Derivation 允许覆盖**：

- Layout：可在派生 Profile 内修改布局配置白名单属性

**统一禁止覆盖**：

- code、type、scope、profile.code 等关键标识字段
- fields、links、indexes 等结构定义
- `customizable=false` 的元数据对象

### 租户扩展能力

**1. 字段可见性控制**：

- 可以禁用平台定义的字段（如果字段的 `customizable=true`）
- 不能新增字段
- 通过 Metadata Customization 覆盖 status 为 DISABLED 实现：
  ```json
  {
    "targetType": "field",
    "targetCode": "internalNotes",
    "properties": {
      "status": "DISABLED"
    }
  }
  ```

**2. 属性扩展**：

- 可以添加租户级别的 Attribute（`scope=TENANT`）
- 可以禁用平台基线的 Attribute（如果属性的 `customizable=true`）
- 属性与字段类似，支持动态扩展
- `composite` 为 Attribute 专用类型，仅用于父属性占位；不属于 Field Type 契约，Field 不可使用
- 通过 Metadata Customization 覆盖 status 为 DISABLED 实现禁用

**3. 属性覆盖**：

- 可以覆盖字段和属性的配置（仅限 `customizable=true` 的元数据）
- 可以通过 Client Customization 覆盖客户端配置（Client）
- 可以通过 LayoutProfile 派生覆盖布局配置（Layout）
- 通过 Metadata Customization 覆盖属性实现：
  ```json
  {
    "targetType": "field",
    "targetCode": "productName",
    "properties": {
      "name": "商品名称",
      "required": false
    }
  }
  ```

**注意**：实体/字段/属性使用统一的状态管理机制，通过 `status` 字段控制启用/禁用状态；Client/Layout 不独立维护 status，由选中配置控制生效。

## 元数据合并机制

元数据合并将平台基线元数据与租户定制配置合并，生成租户专属的元数据视图。详细的技术实现方案见：[../../01.architecture/00.foundation/02.metadata/](../../01.architecture/00.foundation/02.metadata/)

### 合并策略

- **属性覆盖**：通过 Metadata Customization、Client Customization、Layout Profile Derivation 覆盖平台基线配置
- **状态过滤**：过滤 `status=DISABLED` 的元数据对象
- **权限控制**：只合并 `customizable=true` 的对象
- **配置优先级**：Client（Platform < Tenant）；Layout（Platform < Tenant < User）

### 合并流程

1. 实体级别：检查实体的 `status` 和 `customizable` 属性
2. 字段级别：合并字段属性，过滤禁用字段
3. 属性级别：合并属性配置，添加租户自定义属性
4. 配置级别：合并客户端配置和布局配置

## 数据模型关系

```
Entity Definition (1) ──< (N) Field
Entity Definition (1) ──< (N) Link
Entity Definition (1) ──< (N) Index
Entity Definition (1) ──< (N) Attribute
Entity Definition (1) ──< (N) Client
Entity Definition (1) ──< (N) Layout
LayoutProfile (1) ──< (N) Layout
Attribute (1) ──< (N) Attribute (自关联)
MetadataCustomization (N) ──< (1) Entity Definition (通过 entityCode 关联)
TenantClientCustomization (N) ──< (1) Entity Definition (通过 entityCode 关联)
Layout (entityCode+profileId) 绑定 Entity 与 LayoutProfile
Tenant/User 通过派生 LayoutProfile 并在 Layout 层绑定实体完成定制
```
