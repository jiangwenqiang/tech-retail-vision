# Layout Definition - 布局编排定义

## 作用

Layout 定义页面布局和面板编排。

## 核心字段

| 字段         | 类型         | 必填 | 默认值 | 说明                                                        |
| ------------ | ------------ | ---- | ------ | ----------------------------------------------------------- |
| entityCode   | VARCHAR(100) | 是   | -      | 关联实体编码（Entity.code）                                 |
| profileId    | BIGINT       | 是   | -      | 关联 LayoutProfile ID                                       |
| type         | VARCHAR(50)  | 是   | -      | 布局类型                                                    |
| customizable | BOOLEAN      | 是   | true   | 条目修改闸门：是否允许在已派生 Profile 中修改该 Layout 条目 |
| layout       | JSON         | 是   | -      | 布局配置                                                    |
| yn           | CHAR(1)      | 是   | N      | 软删除标记                                                  |

## 布局类型

| 类型     | 代码         | 说明         |
| -------- | ------------ | ------------ |
| 列表布局 | list         | 列表页配置   |
| 详情布局 | detail       | 详情页配置   |
| 表单布局 | form         | 表单配置     |
| 关系布局 | relationship | 关系面板配置 |

### Layout 与 LayoutProfile 关系

- `LayoutProfile` 是布局方案容器（如 `default`、`retail-b2b`）
- `Layout` 是方案下的具体布局条目（按 `type` 与关系维度区分）
- `LayoutProfile` 不绑定实体；同一 Profile 可被多个实体复用
- `Layout` 通过 `entityCode + profileId` 绑定具体实体与具体方案
- `Layout` 不承载作用域；作用域统一由 `LayoutProfile(scope+owner)` 承载
- 唯一性基于 `entityCode + profileId + type`
- 当 `type=relationship` 时，具体关系配置在 `layout.relationshipLayouts` 中按 `linkCode` 分组

### relationship 布局使用场景

`relationship` 用于定义“实体详情页中的关系面板”如何展示，不是主列表或主详情布局的替代。

典型场景：

- 同一实体下，不同关系链接需要不同列配置（如 `categories` 与 `tags`）
- 关系面板需要独立的列宽、排序、默认过滤条件
- 关系面板需要与主页面布局解耦，支持按关系单独调整

边界说明：

- `Layout.relationship` 负责关系面板的编排（字段、顺序、样式）
- `Client.relationshipPanels` 负责关系面板的行为能力（选择、创建、动作、拖拽等）

在 Profile 体系中，关系布局通过 `entityCode + profileId + type=relationship` 定位，具体 link 维度由 `layout.relationshipLayouts` 的 key 表达（例如 `categories`、`tags`）。

`relationshipLayouts` 约束建议：

- key 必须是实体已定义的 `Link.code`
- 每个 key 对应一组独立关系面板配置（列、过滤、动作）

## 布局示例（场景化）

### 场景 1：商品管理列表（list）

```json
{
  "entityCode": "Product",
  "profileId": 9001,
  "type": "list",
  "layout": {
    "columns": [
      { "field": "productName", "width": 260, "sortable": true },
      { "field": "productCode", "width": 180, "sortable": true },
      { "field": "brand", "width": 160 },
      { "field": "status", "width": 110 },
      { "field": "updatedAt", "width": 180, "sortable": true }
    ],
    "filters": [
      { "field": "status", "operator": "equals" },
      { "field": "brand", "operator": "equals" },
      { "field": "updatedAt", "operator": "between" }
    ],
    "actions": ["create", "batchEdit", "export", "delete"]
  }
}
```

### 场景 2：商品详情页（detail）

```json
{
  "entityCode": "Product",
  "profileId": 9001,
  "type": "detail",
  "layout": {
    "panels": [
      {
        "name": "基本信息",
        "rows": [
          [{ "field": "productName" }, { "field": "productCode" }],
          [
            { "field": "brand" },
            { "field": "status" },
            { "field": "updatedAt" }
          ]
        ]
      },
      {
        "name": "销售信息",
        "rows": [
          [{ "field": "price" }, { "field": "currency" }],
          [{ "field": "taxClass" }, { "field": "salesChannel" }]
        ]
      }
    ]
  }
}
```

### 场景 3：商品编辑表单（form）

```json
{
  "entityCode": "Product",
  "profileId": 9001,
  "type": "form",
  "layout": {
    "sections": [
      {
        "name": "基础信息",
        "fields": ["productName", "productCode", "brand", "status"]
      },
      {
        "name": "价格信息",
        "fields": ["price", "currency", "taxClass"]
      }
    ],
    "submitActions": ["save", "saveAndNew"]
  }
}
```

### 场景 4：商品-分类关系面板（relationship）

```json
{
  "entityCode": "Product",
  "profileId": 9001,
  "type": "relationship",
  "layout": {
    "relationshipLayouts": {
      "categories": {
        "columns": [
          { "field": "name", "width": 220, "sortable": true },
          { "field": "code", "width": 140, "sortable": true },
          { "field": "status", "width": 100 }
        ],
        "filters": [{ "field": "status", "operator": "equals" }],
        "actions": ["select", "unlink"]
      },
      "tags": {
        "columns": [
          { "field": "name", "width": 220 },
          { "field": "type", "width": 140 }
        ],
        "actions": ["select", "unlink"]
      }
    }
  }
}
```

## 基线与租户管理

### 平台布局（PLATFORM）

- 平台定义默认布局
- 每个实体至少有一个标准布局方案（取 profile.status = ACTIVE 的数据）
- 可以有多套布局方案（多个 `profile.code`），但是启用的只有1个
- 当前平台维度启用的布局所有租户可见
- 租户不能直接修改平台布局

### Tenant/User 定制（Profile 派生）

- Layout 不独立分层存储 `scope`
- 当 `customizable = true` 时，通过派生 LayoutProfile 定制布局配置
- 运行时：User Profile > Tenant Profile > Platform Profile

### `customizable` 双闸门规则

- 判定顺序：先 `LayoutProfile.customizable`，后 `Layout.customizable`
- 拒绝优先（deny-first）：任一层为 `false`，拒绝对应变更操作
- 语义分工：
  - `LayoutProfile.customizable`：控制是否允许派生下一层 Profile
  - `Layout.customizable`：控制在已派生 Profile 内，该 Layout 条目是否允许修改
- 决策矩阵：
  - `Profile=false, Layout=*`：禁止派生
  - `Profile=true, Layout=false`：允许派生，但条目只读继承
  - `Profile=true, Layout=true`：允许派生且允许修改条目

### 可见性与定制化

- Layout Definition 本身不支持可见性控制
- Layout 不独立维护 `status`，其生效由关联实体状态与选中 Profile 配置共同决定
- 通过实体与 Layout 的 `customizable` 字段共同控制是否允许租户定制布局配置
