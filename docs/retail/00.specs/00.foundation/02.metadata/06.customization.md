# Customization - 定制配置

## 作用

Customization 是一个统一的配置对象，用于管理租户对平台基线元数据的定制化配置。通过通用的目标定位和操作类型，支持对各种元数据对象（Field、Attribute、Client、Layout 等）的定制化。

## 设计原则

1. **通用性**：统一的配置结构，支持所有类型的元数据对象
2. **可扩展性**：易于添加新的操作类型和目标类型
3. **类型安全**：通过 targetType 和操作类型确保配置的合法性
4. **权限控制**：只能定制 `customizable=true` 的元数据对象

## 核心字段

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|-----|------|------|-------|------|
| tenantId | BIGINT | 是 | - | 租户ID |
| entityId | BIGINT | 是 | - | 关联实体ID |
| targetType | VARCHAR(50) | 是 | - | 目标类型（field/attribute/client/layout等） |
| targetCode | VARCHAR(100) | 是 | - | 目标编码（字段编码、属性编码等） |
| operation | VARCHAR(50) | 是 | - | 操作类型（hide/override/delete等） |
| config | JSON | 否 | - | 操作配置（根据 operation 类型不同而不同） |
| sortOrder | INT | 是 | 0 | 排序号（确保配置按顺序应用） |
| yn | CHAR(1) | 是 | N | 软删除标记 |

## 目标类型（targetType）

| 类型 | 代码 | 说明 | 支持的操作 |
|-----|------|------|-----------|
| 字段 | field | 实体字段 | hide, override |
| 属性 | attribute | 实体属性 | hide, override |
| 客户端配置 | client | 客户端配置 | override |
| 布局配置 | layout | 布局配置 | override |

## 操作类型（operation）

| 操作 | 代码 | 说明 | 适用目标 | config 结构 |
|-----|------|------|---------|-------------|
| 隐藏 | hide | 隐藏目标对象 | field, attribute | null |
| 覆盖 | override | 覆盖目标对象属性 | field, attribute, client, layout | 覆盖属性对象 |
| 删除 | delete | 删除目标对象（预留） | attribute | null |

## 配置结构

### hide 操作 - 隐藏字段/属性

```json
{
  "tenantId": 1001,
  "entityId": 100,
  "targetType": "field",
  "targetCode": "internalNotes",
  "operation": "hide",
  "config": null,
  "sortOrder": 1
}
```

### override 操作 - 覆盖字段

```json
{
  "tenantId": 1001,
  "entityId": 100,
  "targetType": "field",
  "targetCode": "productName",
  "operation": "override",
  "config": {
    "name": "商品名称",
    "required": false,
    "readOnly": true
  },
  "sortOrder": 2
}
```

### override 操作 - 覆盖属性

```json
{
  "tenantId": 1001,
  "entityId": 100,
  "targetType": "attribute",
  "targetCode": "color",
  "operation": "override",
  "config": {
    "name": "颜色",
    "required": true
  },
  "sortOrder": 3
}
```

### override 操作 - 覆盖客户端配置

```json
{
  "tenantId": 1001,
  "entityId": 100,
  "targetType": "client",
  "targetCode": "list",
  "operation": "override",
  "config": {
    "views": {
      "list": "views/record/list-custom"
    }
  },
  "sortOrder": 4
}
```

## 操作详解

### hide - 隐藏操作

**作用**：隐藏指定的元数据对象，使其对租户不可见

**适用目标**：field, attribute

**config**：null

**约束**：
- 目标对象必须存在
- 目标对象的 `customizable` 必须为 `true`

### override - 覆盖操作

**作用**：覆盖元数据对象的属性

**适用目标**：field, attribute, client, layout

**config**：包含要覆盖的属性对象

**字段覆盖支持**：
- name (VARCHAR)
- required (BOOLEAN)
- readOnly (BOOLEAN)

**属性覆盖支持**：
- name (VARCHAR)
- required (BOOLEAN)

**客户端配置覆盖**：
- 完整的客户端配置对象

**约束**：
- 目标对象必须存在
- 目标对象的 `customizable` 必须为 `true`
- 覆盖的属性必须在允许覆盖的范围内

### delete - 删除操作（预留）

**作用**：删除指定的元数据对象

**适用目标**：attribute（租户级别的属性）

**config**：null

**约束**：
- 只能删除 `scope=TENANT` 的对象
- 不能删除平台基线对象

## 运行时合并规则

### 实体可见性检查

1. 检查实体的 `status` 字段，只有 `status=ACTIVE` 的实体才可见
2. 检查实体的 `scope` 字段，PLATFORM 作用域的实体对所有租户可见
3. 如果实体可见，继续检查字段和属性

### 字段合并

1. 获取平台定义的所有字段（`scope=PLATFORM`）
2. 获取该租户对该实体的所有 Customization 配置（`targetType=field`）
3. 按 `sortOrder` 顺序应用配置：
   - **hide 操作**：从字段列表中移除该字段（如果 `customizable=true`）
   - **override 操作**：合并配置到字段属性（如果 `customizable=true`）
4. 返回合并后的字段列表

### 属性合并

1. 获取平台定义的所有属性（`scope=PLATFORM`）
2. 获取该租户对该实体的所有 Customization 配置（`targetType=attribute`）
3. 按 `sortOrder` 顺序应用配置：
   - **hide 操作**：从属性列表中移除该属性（如果 `customizable=true`）
   - **override 操作**：合并配置到属性属性（如果 `customizable=true`）
   - **delete 操作**：删除该属性（仅限 `scope=TENANT` 的属性）
4. 添加租户自定义属性（`scope=TENANT`）
5. 返回合并后的属性列表

### 客户端配置合并

1. 获取平台定义的 Client Definition（`scope=PLATFORM`）
2. 查找 `targetType=client` 且 `operation=override` 的配置
3. 如果有配置，合并配置到客户端定义
4. 返回合并后的客户端配置

### 配置优先级

- sortOrder 越小，优先级越高
- 后面的配置会覆盖前面配置的相同属性
- 租户配置 > 平台默认

## 扩展性设计

### 添加新的目标类型

如果需要支持新的元数据对象定制（如 Relationship、Index 等）：

1. 在 `targetType` 枚举中添加新类型
2. 定义该类型支持的操作
3. 定义操作的 config 结构
4. 实现相应的运行时合并逻辑

示例：支持 Relationship 定制

```json
{
  "targetType": "relationship",
  "targetCode": "category",
  "operation": "override",
  "config": {
    "name": "产品分类",
    "required": true
  }
}
```

### 添加新的操作类型

如果需要支持新的操作（如 reorder、duplicate 等）：

1. 在 `operation` 枚举中添加新操作
2. 定义操作的 config 结构
3. 实现相应的运行时逻辑
4. 更新合并规则

示例：支持排序操作

```json
{
  "targetType": "field",
  "targetCode": "productName",
  "operation": "reorder",
  "config": {
    "sortOrder": 10
  }
}
```

```json
    "targetCode": "color",

      "required": true
    },
    "sortOrder": 3
  }

```

## 约束规则

1. **唯一性**：`tenantId + entityId + targetType + targetCode + yn` 必须唯一
2. **配置完整性**：`config` 必须符合操作类型要求的结构
3. **目标存在性**：`targetCode` 必须在实体定义中存在
4. **权限验证**：目标对象的 `customizable` 必须为 `true`
5. **操作合法性**：操作类型必须与目标类型匹配

## 与旧设计的对比

### 旧设计（config 包含所有配置）

```json
{
  "tenantId": 1001,
  "entityId": 100,
  "config": {
    "hiddenFields": ["field1", "field2"],
    "fieldOverrides": {
      "field1": {"name": "新名称"}
    }
  }
}
```

**问题**：
- 需要为每种目标类型定义不同的配置结构
- 扩展性差，添加新目标类型需要修改 schema
- 难以表达复杂的定制逻辑

### 新设计（统一配置条目）

```json
[
  {
    "targetType": "field",
    "targetCode": "field1",
    "operation": "hide"
  },
  {
    "targetType": "field",
    "targetCode": "field2",
    "operation": "override",
    "config": {"name": "新名称"}
  }
]
```

**优势**：
- 统一的数据结构
- 易于扩展新的目标类型和操作
- 每个配置条目独立，便于管理和调试
- 支持 sortOrder 控制应用顺序
