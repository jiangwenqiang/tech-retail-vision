# Customization - 定制配置

## 作用

Customization 是一个统一的配置对象，用于管理租户对平台基线元数据的定制化配置。通过覆盖元数据对象的属性，支持对各种元数据对象（Field、Attribute、Client、Layout 等）的定制化。

## 设计原则

1. **通用性**：统一的配置结构，支持所有类型的元数据对象
2. **可扩展性**：易于添加新的目标类型
3. **状态一致**：通过 `status` 字段统一控制元数据对象的启用/禁用
4. **权限控制**：只能定制 `customizable=true` 的元数据对象

## 核心字段

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|-----|------|------|-------|------|
| tenantId | BIGINT | 是 | - | 租户ID |
| entityId | BIGINT | 是 | - | 关联实体ID |
| targetType | VARCHAR(50) | 是 | - | 目标类型（field/attribute/client/layout等） |
| targetCode | VARCHAR(100) | 是 | - | 目标编码（字段编码、属性编码等） |
| properties | JSON | 是 | - | 要覆盖的属性对象 |
| sortOrder | INT | 是 | 0 | 排序号（确保配置按顺序应用） |
| yn | CHAR(1) | 是 | N | 软删除标记 |

## 目标类型（targetType）

| 类型 | 代码 | 说明 | 可覆盖的属性 |
|-----|------|------|------------|
| 字段 | field | 实体字段 | name, required, readOnly, status |
| 属性 | attribute | 实体属性 | name, required, status |
| 客户端配置 | client | 客户端配置 | 完整的客户端配置对象 |
| 布局配置 | layout | 布局配置 | 完整的布局配置对象 |

## 配置结构

### 禁用字段（通过 status）

```json
{
  "tenantId": 1001,
  "entityId": 100,
  "targetType": "field",
  "targetCode": "internalNotes",
  "properties": {
    "status": "DISABLED"
  },
  "sortOrder": 1
}
```

### 覆盖字段属性

```json
{
  "tenantId": 1001,
  "entityId": 100,
  "targetType": "field",
  "targetCode": "productName",
  "properties": {
    "name": "商品名称",
    "required": false,
    "readOnly": true
  },
  "sortOrder": 2
}
```

### 覆盖属性配置

```json
{
  "tenantId": 1001,
  "entityId": 100,
  "targetType": "attribute",
  "targetCode": "color",
  "properties": {
    "name": "颜色",
    "required": true
  },
  "sortOrder": 3
}
```

### 覆盖客户端配置

```json
{
  "tenantId": 1001,
  "entityId": 100,
  "targetType": "client",
  "targetCode": "list",
  "properties": {
    "views": {
      "list": "views/record/list-custom"
    }
  },
  "sortOrder": 4
}
```

## 属性覆盖详解

### 状态控制（status）

通过覆盖 `status` 属性来控制元数据对象的启用/禁用状态：

| 状态 | 说明 | 效果 |
|-----|------|------|
| ACTIVE | 启用 | 元数据对象正常可用 |
| DISABLED | 禁用 | 元数据对象对租户不可见 |
| DRAFT | 草稿 | 元数据对象未生效（用于租户自定义属性） |

**示例：禁用字段**
```json
{
  "targetType": "field",
  "targetCode": "internalNotes",
  "properties": {
    "status": "DISABLED"
  }
}
```

### 字段属性覆盖

**支持覆盖的属性**：
- name (VARCHAR) - 字段名称
- required (BOOLEAN) - 是否必填
- readOnly (BOOLEAN) - 是否只读
- status (VARCHAR) - 启用状态

**约束**：
- 目标字段必须存在
- 目标字段的 `customizable` 必须为 `true`
- 不能覆盖 code、type、scope 等关键字段

### 属性配置覆盖

**支持覆盖的属性**：
- name (VARCHAR) - 属性名称
- required (BOOLEAN) - 是否必填
- status (VARCHAR) - 启用状态

**约束**：
- 目标属性必须存在
- 目标属性的 `customizable` 必须为 `true`
- 不能覆盖 code、type、scope 等关键字段

### 客户端配置覆盖

**支持覆盖**：
- 完整的客户端配置对象

**约束**：
- 实体的 `customizable` 必须为 `true`
- 对应的 Client Definition 的 `customizable` 必须为 `true`

### 布局配置覆盖

**支持覆盖**：
- 完整的布局配置对象

**约束**：
- 实体的 `customizable` 必须为 `true`
- 对应的 Layout 的 `customizable` 必须为 `true`

## 运行时合并规则

租户定制配置需要与平台基线元数据合并，以生成租户专属的元数据视图。详细的技术实现方案见：[../../01.architecture/00.foundation/02.metadata/](../../01.architecture/00.foundation/02.metadata/)

### 合并原则

- **权限验证**：只能定制 `customizable=true` 的元数据对象
- **属性限制**：不能覆盖 code、type、scope 等关键字段
- **状态优先**：`status=DISABLED` 的对象不会出现在合并结果中
- **排序保证**：按 `sortOrder` 顺序应用定制配置
- **配置优先级**：租户配置 > 平台默认

### 合并流程

1. **实体级别**：检查实体的 `status` 和 `customizable` 属性
2. **字段级别**：合并字段属性，过滤 `status=DISABLED` 的字段
3. **属性级别**：合并属性配置，添加租户自定义属性
4. **配置级别**：合并客户端配置和布局配置

### 合并示例

**平台基线**：
```json
{
  "fields": [
    {"code": "productName", "name": "Product Name", "required": true},
    {"code": "internalNotes", "name": "Internal Notes", "customizable": true}
  ]
}
```

**租户定制**：
```json
[
  {
    "targetType": "field",
    "targetCode": "productName",
    "properties": {"name": "商品名称"}
  },
  {
    "targetType": "field",
    "targetCode": "internalNotes",
    "properties": {"status": "DISABLED"}
  }
]
```

**合并结果**：
```json
{
  "fields": [
    {"code": "productName", "name": "商品名称", "required": true}
    // internalNotes 被 status=DISABLED 过滤掉
  ]
}
```

## 扩展性设计

### 添加新的目标类型

如果需要支持新的元数据对象定制（如 Relationship、Index 等）：

1. 在 `targetType` 枚举中添加新类型
2. 定义该类型支持覆盖的属性
3. 实现相应的运行时合并逻辑

示例：支持 Relationship 定制

```json
{
  "targetType": "relationship",
  "targetCode": "category",
  "properties": {
    "name": "产品分类",
    "required": true
  }
}
```

### 添加新的可覆盖属性

如果需要支持覆盖更多的属性：

1. 在目标类型的可覆盖属性列表中添加新属性
2. 更新权限验证逻辑
3. 更新合并规则

示例：支持 Field 的默认值覆盖

```json
{
  "targetType": "field",
  "targetCode": "status",
  "properties": {
    "defaultValue": "ACTIVE"
  }
}
```

## 约束规则

1. **唯一性**：`tenantId + entityId + targetType + targetCode + yn` 必须唯一
2. **配置完整性**：`properties` 必须是有效的 JSON 对象
3. **目标存在性**：`targetCode` 必须在实体定义中存在
4. **权限验证**：目标对象的 `customizable` 必须为 `true`
5. **属性合法性**：`properties` 中的属性必须在可覆盖的范围内
6. **状态一致性**：覆盖 `status` 时，必须是有效的状态值（ACTIVE/DISABLED/DRAFT）

## 与旧设计的对比

### 旧设计（config 包含所有配置）

```json
{
  "tenantId": 1001,
  "entityId": 100,
  "config": {
    "hiddenFields": ["field1", "field2"],
    "fieldOverrides": {
      "field1": {"name": "新名称"}
    }
  }
}
```

**问题**：
- 需要为每种目标类型定义不同的配置结构
- 扩展性差，添加新目标类型需要修改 schema
- 难以表达复杂的定制逻辑
- hide 操作与 status 字段功能重复

### 新设计（统一属性覆盖）

```json
[
  {
    "targetType": "field",
    "targetCode": "field1",
    "properties": {
      "status": "DISABLED"
    }
  },
  {
    "targetType": "field",
    "targetCode": "field2",
    "properties": {"name": "新名称"}
  }
]
```

**优势**：
- 统一的数据结构
- 易于扩展新的目标类型
- 每个配置条目独立，便于管理和调试
- 通过 status 字段统一控制启用/禁用状态
- 支持 sortOrder 控制应用顺序
- 与元数据对象的状态管理机制保持一致
