# Scope Permission - 范围权限

## 作用

Scope Permission 统一控制"实体读权限通过后，用户可见哪些记录"。所有记录级过滤条件——无论是组织隔离、所有权限定还是自定义业务条件——都通过同一个 ScopePermission 实体表达。


## 核心字段

| 字段 | 类型 | 必填 | 说明 |
|-----|------|------|------|
| entityCode | VARCHAR(100) | 是 | 实体编码（Entity.code） |
| roleCode | VARCHAR(100) | 是 | 角色编码（Role.code） |
| filterCode | VARCHAR(100) | 是 | 过滤器编码（实体内唯一） |
| name | VARCHAR(255) | 是 | 过滤器名称 |
| type | VARCHAR(20) | 是 | 过滤器类型 |
| props | JSON | 否 | 过滤参数 |
| isEnabled | BOOLEAN | 是 | true | 是否启用 |
| sortOrder | INT | 是 | 0 | 排序号（执行顺序） |
| yn | CHAR(1) | 是 | N | 软删除标记 |

## 过滤器类型（type）

| 类型 | 代码 | 说明 | 适用场景 |
|-----|------|------|---------|
| 范围 | scope | 预定义的组织级/所有权级过滤 | 组织隔离、我的数据 |
| 字段条件 | field | 基于实体字段的等值/范围比较 | 按状态、分类过滤 |
| 表达式 | expression | 组合条件表达式 | 多字段组合过滤 |
| 回调 | callback | 后端自定义逻辑 | 复杂业务规则 |

**统一语义**：无论哪种 type，最终都产生一个 WHERE 条件。多个 ScopePermission 按 sortOrder 排序后全部 AND 拼接。

```
SELECT * FROM `order`
WHERE tenant_id = ? AND yn = 'N'
  AND organization_id IN (?, ?, ?)      ← type=scope
  AND status NOT IN ('Cancelled')       ← type=field
  AND created_at >= ?                   ← type=field
```

## type=scope（范围过滤器）

范围过滤器是预定义的通用数据隔离模式，适用于所有实体。

### props

| 参数 | 类型 | 必填 | 说明 |
|-----|------|------|------|
| level | VARCHAR(20) | 是 | 范围级别 |

### 范围级别

| 级别 | 代码 | 说明 | 零售场景 |
|-----|------|------|---------|
| 全部 | all | 租户内所有记录 | 总经理、区域经理 |
| 组织 | organization | 所属组织及其下级组织的记录 | 门店经理 |
| 自己 | own | 自己创建或分配给自己的记录 | 销售员 |
| 无 | none | 不可见任何记录 | 配置异常时应避免 |

### 范围层级

```
all
 └── organization（所属组织 + 下级组织）
      └── own（自己创建 + 分配给自己）
           └── none（不可见）
```

上级范围天然包含下级范围。`all` 包含 `organization` 包含 `own`。

### 配置示例

```json
{
  "entityCode": "Product",
  "roleCode": "STORE_MANAGER",
  "filterCode": "product_scope",
  "name": "门店产品范围",
  "type": "scope",
  "props": { "level": "organization" },
  "sortOrder": 1
}
```

```json
{
  "entityCode": "Order",
  "roleCode": "SALESMAN",
  "filterCode": "order_scope",
  "name": "我的订单",
  "type": "scope",
  "props": { "level": "own" },
  "sortOrder": 1
}
```

### 组织级隔离（level=organization）

通过 Organization 实体的层级关系实现数据隔离：

```
零售集团（Organization A）
├── 华东区（Organization B）
│   ├── 上海旗舰店（Organization C）
│   └── 杭州西湖店（Organization D）
└── 华南区（Organization E）
    ├── 深圳万象城店（Organization F）
    └── 广州天河店（Organization G）
```

可见性计算流程：

```
1. 获取用户所属组织
       ↓
2. 递归查询所有下级组织
       ↓
3. 可见组织集合 = [所属组织] ∪ [所有下级组织]
       ↓
4. 过滤记录 WHERE organization_id IN (可见组织集合)
```

层级遍历示例：

```
叶子节点用户（张三）绑定到 Organization C（上海旗舰店）
  → 可见组织集合 = [C]
  → WHERE organization_id IN ('C')

中间节点用户（李四）绑定到 Organization B（华东区）
  → 可见组织集合 = [B, C, D]
  → WHERE organization_id IN ('B', 'C', 'D')

根节点用户（王五）绑定到 Organization A（零售集团）
  → 可见组织集合 = [A, B, C, D, E, F, G]
  → WHERE organization_id IN ('A', 'B', 'C', 'D', 'E', 'F', 'G')
```

业务记录通过 `organizationId` 字段关联到组织：

| 字段 | 类型 | 说明 |
|-----|------|------|
| organizationId | VARCHAR(100) | 记录归属的组织 ID |
| assignedUserId | VARCHAR(100) | 记录分配给的用户 ID |
| createdById | VARCHAR(100) | 记录创建者用户 ID |

用户可绑定多个组织，此时可见组织集合为各组织的并集：

```
用户（赵六）绑定到 Organization C 和 Organization F
  → 可见组织集合 = [C, F]
  → WHERE organization_id IN ('C', 'F')
```

组织变动的影响：

| 变动 | 影响范围 | 说明 |
|-----|---------|------|
| 组织创建 | 该组织父节点的所有成员 | 新增的可见组织立即生效 |
| 组织删除 | 该组织及其下级的所有成员 | 需要先处理下级组织 |
| 组织移动（换父节点） | 旧父节点和新父节点的成员 | 可见集合重新计算 |

### 所有权过滤（level=own）

满足以下任一条件的记录可见：

| 条件 | 字段 |
|-----|------|
| 创建者是本人 | `createdById = currentUserId` |
| 分配给本人 | `assignedUserId = currentUserId` |

分配规则：记录的 `assignedUserId` 变更受 EntityPermission(type=edit) 约束：

| EntityPermission.edit | 可分配范围 |
|----------------------|-----------|
| level=yes | 可分配给租户内任意用户 |
| level=own + allowAssignTo=self | 可分配给自己 |
| level=own + allowAssignTo=none | 不可更改分配关系 |
| level=no | 不可编辑，不可分配 |

分配后的可见性变化：

```
销售员 A 创建订单 → assignedUserId = A
  ↓
店长 B 将订单分配给销售员 C → assignedUserId = C
  ↓
销售员 A（scope=own）→ 仍可见（createdById = A）
销售员 C（scope=own）→ 变为可见（assignedUserId = C）
```

### 运行时 SQL 注入

| level | 注入条件 |
|-------|---------|
| all | 不注入额外条件 |
| organization | `AND organization_id IN (?, ?, ?)` |
| own | `AND (created_by = ? OR assigned_user_id = ?)` |
| none | `WHERE 1 = 0`（返回空结果） |

### scope 与 edit/delete level=own 的关系

scope 决定"可见哪些记录"，edit/delete 的 level=own 在可见记录集上进一步缩小操作范围：

```
ScopePermission(type=scope, level=organization)
  → 可见 [B, C, D] 组织的记录
    → EntityPermission(type=edit, level=own)
      → 仅编辑其中 createdById/assignedUserId = 自己的记录
    → EntityPermission(type=edit, level=yes)
      → 可编辑其中所有记录
```

### 无 scope 过滤器的实体

部分系统级实体不按组织或所有权隔离：

| 实体类型 | 建议配置 | 说明 |
|---------|---------|------|
| ReferenceData（字典） | 不配置 scope 过滤器 | 全租户共享 |
| Archive（归档） | 不配置 scope 过滤器 | 系统记录 |
| Hierarchy（层级实体） | type=scope, level=organization | 按组织隔离 |
| Base（基础实体） | 视业务而定 | 由配置决定 |

不配置 scope 过滤器时，等同于 level=all（租户内所有记录可见）。

### 约束

- 每个角色在每个实体上**建议只有一个** type=scope 的过滤器
- 若存在多个 type=scope 的过滤器，取 level 最宽的（`all > organization > own > none`）
- `level=organization` 时，用户必须绑定至少一个有效组织
- `level=own` 依赖 `createdById` 和 `assignedUserId` 字段，实体应包含这两个字段
- 组织层级不能形成环（由 Organization 实体自身保证）

## type=field（字段条件过滤器）

基于实体字段的等值/范围比较。

### props

```json
{
  "conditions": [
    {
      "field": "status",
      "operator": "notIn",
      "value": ["Cancelled", "Completed"]
    }
  ]
}
```

### 支持的 operator

| 操作符 | 说明 | 示例 |
|-------|------|------|
| equals | 等于 | `{ "field": "status", "operator": "equals", "value": "Active" }` |
| notEquals | 不等于 | `{ "field": "status", "operator": "notEquals", "value": "Draft" }` |
| in | 在列表中 | `{ "field": "status", "operator": "in", "value": ["Active", "Pending"] }` |
| notIn | 不在列表中 | `{ "field": "status", "operator": "notIn", "value": ["Cancelled"] }` |
| greaterThan | 大于 | `{ "field": "amount", "operator": "greaterThan", "value": 1000 }` |
| lessThan | 小于 | `{ "field": "createdAt", "operator": "lessThan", "value": "2024-01-01" }` |
| isNull | 为空 | `{ "field": "assignedUserId", "operator": "isNull" }` |
| isNotNull | 不为空 | `{ "field": "assignedUserId", "operator": "isNotNull" }` |

### 配置示例

```json
{
  "entityCode": "Order",
  "roleCode": "SALESMAN",
  "filterCode": "active_orders_only",
  "name": "仅进行中的订单",
  "type": "field",
  "props": {
    "conditions": [
      { "field": "status", "operator": "notIn", "value": ["Cancelled", "Completed"] }
    ]
  },
  "sortOrder": 2
}
```

## type=expression（表达式过滤器）

通过逻辑组合多个条件。

### props

```json
{
  "logic": "AND",
  "conditions": [
    { "field": "primaryCategoryCode", "operator": "in", "value": ["${user.managedCategories}"] }
  ]
}
```

### 支持的 logic

| 逻辑 | 说明 |
|-----|------|
| AND | 所有条件必须满足 |
| OR | 任一条件满足即可 |

### 配置示例

```json
{
  "entityCode": "Product",
  "roleCode": "CATEGORY_MANAGER",
  "filterCode": "assigned_category_only",
  "name": "仅负责的品类",
  "type": "expression",
  "props": {
    "logic": "AND",
    "conditions": [
      { "field": "primaryCategoryCode", "operator": "in", "value": ["${user.managedCategories}"] }
    ]
  },
  "sortOrder": 2
}
```

## type=callback（回调过滤器）

通过后端代码实现自定义过滤逻辑。

### props

```json
{
  "handler": "order.recentOnly",
  "options": {
    "days": 30
  }
}
```

### 配置示例

```json
{
  "entityCode": "Order",
  "roleCode": "SALESMAN",
  "filterCode": "recent_orders",
  "name": "仅最近30天订单",
  "type": "callback",
  "props": {
    "handler": "order.recentOnly",
    "options": { "days": 30 }
  },
  "sortOrder": 3
}
```

后端实现（伪代码）：

```typescript
registerHandler("order.recentOnly", (query, options, context) => {
  const since = new Date()
  since.setDate(since.getDate() - options.days)
  query.where("created_at", ">=", since)
})
```

## 运行时行为

### 执行顺序

ScopePermission 在 EntityPermission(type=read, level=yes) 之后生效，多个过滤器按 `sortOrder` 排序后全部 AND 拼接：

```
原始查询
  ↓
EntityPermission 检查（type=read, level=yes）
  ↓
ScopePermission（按 sortOrder 依次注入 WHERE 条件）
  ↓
EntityPermission（type=edit/delete, level=yes/own）→ 在可见记录集上操作
  ↓
FieldPermission → 字段裁剪
  ↓
返回结果
```

### 执行规则

1. **按 sortOrder 排序**：sortOrder 越小越先执行
2. **全部 AND 拼接**：所有启用的过滤器条件取交集
3. **type=scope 优先**：建议 sortOrder=1，确保组织/所有权过滤最先注入（性能最优，命中索引）

## 零配置默认值

当角色没有为某个实体配置任何 ScopePermission 时：

| 场景 | 行为 |
|-----|------|
| 无任何过滤器 | 不追加过滤条件，等同于 type=scope level=all |
| 有过滤器但全部禁用 | 等同于无过滤器 |

建议为每个角色-实体组合至少配置一个 type=scope 的过滤器，明确声明数据可见范围。

## 多角色合并

当用户绑定多个角色时，ScopePermission 取**交集**——所有角色的过滤器条件都会追加：

```
角色 A: type=scope, level=organization  →  WHERE organization_id IN (B, C, D)
角色 B: type=field, status NOT IN ('Cancelled')  →  AND status NOT IN ('Cancelled')
合并: WHERE organization_id IN (B, C, D) AND status NOT IN ('Cancelled')
```

**type=scope 合并**：多个角色的 scope 过滤器取最宽 level（`all > organization > own > none`）。

**其他 type 合并**：所有角色的非 scope 过滤器全部保留，AND 拼接。


|-------------|-------|------|
| Team | ScopePermission(type=scope, level=organization) | 统一为 ScopePermission |
| onlyMy | ScopePermission(type=scope, level=own) | 统一为 ScopePermission |
| Boolean Filter | ScopePermission(type=field/expression/callback) | 统一为 ScopePermission |

## 约束规则

1. **实体存在性**：`entityCode` 必须引用有效的 `Entity.code`
2. **角色存在性**：`roleCode` 必须引用有效的 `Role.code`
3. **编码唯一性**：`entityCode + filterCode + yn` 必须唯一
4. **type 值域**：`type` 必须为 `scope`、`field`、`expression`、`callback` 之一
5. **参数契约**：`props` 必须符合对应 `type` 的参数白名单
6. **回调注册**：`type=callback` 时，`props.handler` 必须在系统中有注册

## 扩展点

| 方向 | 说明 |
|-----|------|
| 跨实体过滤 | 支持 conditions 中引用关联实体字段 |
| 动态参数 | 支持 `${user.*}`、`${context.*}` 等运行时变量 |
| 过滤器模板 | 基于实体的 PrimaryFilter 自动生成 ScopePermission |
| 自定义 scope | 注册新的 scope level handler（如 `region`、`channel`） |
