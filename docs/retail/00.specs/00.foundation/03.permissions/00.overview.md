# 权限模型

面向零售行业的多租户权限模型。本方案在现有元数据授权架构（RBAC + ABAC）基础上，扩展为覆盖业务实体的完整权限控制体系。

## 权限分层

```
┌──────────────────────────────────────────────────────────┐
│                     Platform Layer                        │
│           平台级角色（平台管理员/架构师/开发者）              │
│         控制范围：元数据管理、租户管理、系统配置              │
├──────────────────────────────────────────────────────────┤
│                     Entity Layer                          │
│         EntityPermission（type + level）                   │
│         控制范围：能不能对这个实体执行 CRUD                 │
├──────────────────────────────────────────────────────────┤
│                     Record Layer                          │
│         ScopePermission（type=scope/field/expression/callback）│
│         控制范围：实体读权限通过后，可见哪些记录             │
├──────────────────────────────────────────────────────────┤
│                     Field Layer                           │
│         FieldPermission（type + level + props）            │
│         控制范围：同一记录内，可访问哪些字段                 │
└──────────────────────────────────────────────────────────┘
```

**检查优先级**（从外到内）：

1. **平台角色检查** — 平台操作权限（详见 [平台权限](./05.platform-permission.md)）
2. **实体权限检查** — `EntityPermission(type=read, level=yes/no)`
3. **记录过滤** — `ScopePermission`（scope/field/expression/callback 全部 AND 拼接）
4. **字段权限过滤** — `FieldPermission`（字段可见性与可写性）

任一层拒绝即终止，不进入下一层。

## 核心对象

| 对象 | 作用 | 详细文档 |
|-----|------|---------|
| Role Definition | 角色定义 | [01.role.md](./01.role.md) |
| Entity Permission | 实体级操作权限（能不能操作，含审批） | [02.entity-permission.md](./02.entity-permission.md) |
| Scope Permission | 范围权限（能操作哪些记录） | [03.scope-permission.md](./03.scope-permission.md) |
| Field Permission | 字段级权限（能看到哪些字段） | [04.field-permission.md](./04.field-permission.md) |
| Platform Permission | 平台系统操作权限（元数据/租户/系统管理） | [05.platform-permission.md](./05.platform-permission.md) |

## 与现有授权架构的关系

现有授权架构（`01.architecture/00.foundation/02.metadata/04.security/authorization.md`）定义了：

- **平台角色**：`PLATFORM_ADMIN`、`PLATFORM_ARCHITECT`、`PLATFORM_DEVELOPER`（固定角色）
- **元数据权限**：`metadata:entity:create` 等格式（资源:操作）
- **多租户隔离**：基于 `tenantId` 的应用层隔离

本方案在其基础上扩展：

| 维度 | 现有授权 | 本方案扩展 |
|-----|---------|-----------|
| 角色管理 | 6 个固定角色 | 支持租户自定义业务角色 |
| 权限粒度 | 元数据资源级 | 扩展到业务实体级（Product、Order 等） |
| 数据隔离 | 租户级（tenantId） | 细化到组织级（Organization）和用户级 |
| 字段控制 | 无 | 字段级读写/脱敏权限 |
| 记录过滤 | 无 | ScopePermission 统一管理（组织隔离 + 自定义条件） |
| 权限模式 | `resource:action` | 保持一致，扩展资源范围 |

## 权限判定流程

```
用户请求访问 Product 列表
        │
        ▼
  ┌─────────────┐
  │ 平台角色检查  │ ── 否 ──→ 返回 403
  └─────┬───────┘
        │ 是（平台角色拥有所有租户内权限）
        ▼
  ┌─────────────────┐
  │ 实体权限检查      │ ── level=no ──→ 返回 403
  │ type=read        │
  └─────┬───────────┘
        │ level=yes
        ▼
  ┌─────────────────────────────────────────┐
  │ ScopePermission（记录过滤）                   │
  │ ┌─────────────────────────────────────┐ │
  │ │ type=scope:  organization_id IN (...)│ │
  │ │ type=field:  status NOT IN (...)     │ │
  │ │ type=callback: created_at >= ...     │ │
  │ │                   全部 AND 拼接       │ │
  │ └─────────────────────────────────────┘ │
  └─────┬───────────────────────────────────┘
        │ 匹配记录
        ▼
  ┌─────────────┐
  │ 字段权限过滤  │ ── 脱敏/隐藏 ──→ 返回部分字段
  └─────┬───────┘
        │ 允许字段
        ▼
     返回结果
```

## 权限继承与合并

### 多角色合并

一个用户可绑定多个角色，合并规则：

| 对象 | 合并策略 | 说明 |
|-----|---------|------|
| EntityPermission | 按 type 取并集 | 任一角色允许即允许（yes > own > no） |
| ScopePermission(type=scope) | 取最宽 level | all > organization > own > none |
| ScopePermission(其他 type) | 全部保留并 AND 拼接 | 取交集 |
| FieldPermission | 按 type 取并集 | 任一角色允许即允许（yes > masked > no） |

### 平台角色穿透

平台角色拥有对租户内所有数据的完全访问权限（不需要 ScopePermission 和 FieldPermission），不需要单独配置实体权限。平台角色的权限检查独立于租户角色体系。

### 租户管理员

租户管理员（`TENANT_ADMIN`）拥有租户内所有实体权限，但仍受 ScopePermission 和 FieldPermission 约束（除非显式配置为允许）。
