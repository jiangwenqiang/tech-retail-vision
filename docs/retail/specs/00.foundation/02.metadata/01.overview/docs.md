# 总览与设计边界

## 背景与问题

在零售 SaaS 场景中，主数据模型既要平台统一，又要支持租户差异化。若缺少统一元数据体系，常见问题包括：

1. 结构定义分散在代码、数据库、页面配置中，变更不可追踪。
2. 平台升级与租户个性化冲突，导致发布风险高。
3. 字段、属性、布局缺少统一契约，运行时行为不一致。

本方案以“平台基线 + 租户扩展 + 版本治理”为主线，构建统一元数据体系。

## 目标

1. 统一建模：实体、字段类型、字段、属性、客户端布局使用同一元数据语言。
2. 统一存储：元数据持久化到 MySQL，结构前缀统一为 `metadata_`。
3. 统一契约：`field` 与 `field-type` 采用参数契约模型，减少实现歧义。
4. 统一治理：支持版本审计、发布门禁、回滚与灰度控制。
5. 统一扩展：支持租户按范围覆盖与能力提升（promotion）。

## 非目标

1. 不定义具体业务对象字段全集（例如商品、供应商的业务细节）。
2. 不覆盖应用配置域（如菜单、主题、运营参数等）。
3. 不在本章展开数据库 DDL 与流程细节（详见后续章节）。

## 设计原则

1. 基线优先：平台先定义可复用基线，租户在边界内扩展。
2. 显式契约：类型、参数、约束、可见范围必须显式声明。
3. 向后兼容：存量租户运行稳定优先于模型“理想形态”。
4. 可审计：关键变更必须可追踪到版本、人、时间、发布单。
5. 分层解耦：建模层、扩展层、存储层、治理层职责清晰。

## 分层模型（概览）

1. `02.baseline-metadata`：平台基线模型，定义标准元数据资产。
2. `03.tenant-metadata`：租户扩展模型，定义可见范围、覆盖与提升规则。
3. `04.runtime-storage`：运行时与存储模型，定义落库结构和读写流。
4. `05.governance`：生命周期治理，定义版本、发布、回滚与审计机制。

## 核心对象与关系

1. `entity`：业务对象边界，承载字段与属性。
2. `field-type`：字段类型模板，定义参数集合与默认行为。
3. `field`：实体结构字段，按 `field-type` 参数契约实例化。
4. `attribute`：业务语义扩展单元，支持复合属性结构。
5. `layout`：客户端布局组织模型，负责展示编排，不承载业务值。

关键关系：

1. `entity -> field`
2. `field -> field-type`（参数契约关系）
3. `entity -> attribute`
4. `layout -> field/attribute`（仅展示引用关系）

## 运行与存储约束

1. 元数据主存储为 MySQL。
2. 表前缀统一为 `metadata_`。
3. 主键统一为 `BIGINT`。
4. 软删除字段统一 `yn`（`N=有效`, `Y=删除`）。
5. 属性值采用独立值表体系，与结构元数据解耦。

## 生命周期主流程（概览）

1. 建模：平台维护基线模型。
2. 扩展：租户按范围声明可见与覆盖。
3. 验证：发布前执行契约校验与兼容性检查。
4. 发布：版本化下发到运行环境。
5. 观测：记录审计日志与运行指标。
6. 回滚：异常时按版本快速回退。
