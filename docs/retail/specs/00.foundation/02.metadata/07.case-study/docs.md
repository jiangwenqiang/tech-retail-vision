# 案例说明

本文用一个“商品主数据扩展”案例，说明本方案如何从基线定义走到租户生效，并在异常时回滚。

## 案例背景

某零售租户 `tenant_a` 需要在商品（`Product`）中新增“保质期管理”能力，要求：

1. 新增字段：`shelfLifeDays`（保质期天数，整数）。
2. 新增属性组：`storagePolicy`（储存策略），含 `temperatureZone` 与 `lightSensitive`。
3. 在商品编辑页的“基础信息”区域展示上述字段与属性。
4. 本变更仅对 `tenant_a` 生效，不影响其他租户。

## 步骤 1：定义平台基线（可复用部分）

先在基线层定义可复用能力：

1. `field-type`：复用 `int`（参数契约：`min/max/default`）。
2. `attribute`：定义 `storagePolicy` 复合属性结构。
3. `layout`：定义可引用的区块槽位（不绑定租户）。

说明：基线层只定义“能力模板”，不直接声明某租户是否启用。

## 步骤 2：租户扩展与覆盖

在 `03.tenant-metadata` 中为 `tenant_a` 声明扩展：

1. 对 `Product` 新增字段实例 `shelfLifeDays`，`type=int`，参数为 `min=0,max=3650`。
2. 启用属性 `storagePolicy`，并将其可见范围限定为 `tenant_a`。
3. 在 `layout` 中把字段和属性挂入“基础信息”分组。

关键校验：

1. 字段参数必须符合 `int` 类型契约，不允许未声明参数。
2. `layout` 只能引用已存在且当前租户可见的 `field/attribute`。
3. 覆盖策略不得破坏必填、唯一等强约束的兼容性规则。

## 步骤 3：发布与生效

在 `05.governance` 执行版本发布：

1. 生成版本：`metadata_version = v2026.03.14.01`。
2. 执行发布前校验：契约、引用完整性、范围规则。
3. 发布到 `tenant_a`，写入运行时存储（`04.runtime-storage`）。
4. 记录审计：变更人、发布时间、影响对象、变更摘要。

生效结果：

1. `tenant_a` 可在商品页面维护保质期与储存策略。
2. 其他租户无感知，不出现字段与页面污染。

## 步骤 4：异常回滚

若发布后发现前端依赖未就绪，可按版本回滚：

1. 将 `tenant_a` 生效版本回退到上一个稳定版本。
2. 运行时读取恢复旧快照。
3. 保留本次失败版本与审计记录，供后续修复复发。

## 案例中的设计收益

1. 结构与展示分离：`layout` 只编排展示，不承载业务值。
2. 契约可控：字段参数由 `field-type` 管控，降低运行期错误。
3. 租户隔离：扩展按范围生效，避免全局副作用。
4. 可治理：发布、审计、回滚形成闭环。

## 与章节映射

1. 基线定义：`02.baseline-metadata`
2. 租户覆盖：`03.tenant-metadata`
3. 存储落地：`04.runtime-storage`
4. 发布回滚：`05.governance`
